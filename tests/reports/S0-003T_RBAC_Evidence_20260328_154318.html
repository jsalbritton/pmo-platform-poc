Security Observations

OBS-001 · ml_feature_requests INSERT policy uses roles=public instead of roles=authenticated

The INSERT RLS policy on ml_feature_requests specifies roles={public} rather than roles={authenticated}. The WITH CHECK expression (rls_user_role() = ANY(ARRAY['pm','sponsor','admin','ml_admin'])) correctly blocks unauthenticated callers since rls_user_role() returns NULL for anon. However, best practice is to also declare roles={authenticated} to reject attempts at the policy binding layer before evaluating the WITH CHECK. Security posture is functionally correct; recommend updating for defense-in-depth before Sprint 2A production build.

OBS-002 · Test users seeded with PmoTest2026! — must be rotated before production

Six test user accounts have been provisioned with the shared password PmoTest2026! for automated RBAC testing. These accounts must be deactivated or have passwords rotated before the platform is handed off to the consulting firm or promoted to production. Test account credentials must not appear in production user stores. Track as a pre-production checklist item.

OBS-003 · ml_admin role ml_model_versions write not directly REST-tested (no Sprint 2A UI yet)

The ml_model_versions write policy (admin + ml_admin only) was validated at the SQL/RLS layer in Sprint 0 pre-tests. The REST API path for model version writes will be tested in S2A-007T when the model version timeline UI is implemented. This is not a gap — the RLS policy exists and is enforced. TC-010 confirms the SECURITY DEFINER RPC layer is separately protected.
